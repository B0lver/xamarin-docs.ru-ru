# <a name="introduction-to-game-development-with-xamarin"></a>Введение в разработку игр с помощью Xamarin

Разработка игр может оказаться очень сложное, особенно учитывая, как легко можно публикация приложения на мобильных платформах. В этой статье рассматриваются основные понятия и технологии, относящиеся к разработке игр, которые помогут вам создавать игры, ли вашей целью является создание AAA высококачественные игры или просто для программы для интереса.

В этой статье рассматриваются следующие темы:

- **Игры и основные понятия программирования не игры** — мы рассмотрим некоторые основные понятия, которые могут быть уникальными для разработки игр или являются общими с другими типами, разработки, но заслуживают внимания из-за их важности.
- **Команда разработчиков игр** — в этом разделе рассматриваются различные роли в группе разработчиков игр.
- **Создание игр идея** — в этом разделе помогут вам создать новую идею игр — первый шаг в создании новую игру.
- **Технология разработки игр** — здесь будут перечислены некоторые из доступных кросс платформенных технологий, которые могут повысить свою производительность как разработчик игр.


# <a name="game-vs-non-game-programming-concepts"></a>Игры vs. Основные понятия программирования не игры

Программисты, перемещение в разработке игр, часто приходится сталкиваться с новые концепции и шаблоны разработки. В этом разделе представлен высокоуровневый обзор некоторых из этих концепций.


## <a name="the-game-loop"></a>Цикл игры

Типовой игры требуется постоянное перемещение или изменение происходит в ответ на взаимодействие с пользователем и автоматической логику игры на экране. Это достигается с помощью что обычно называется *цикла игры*. Цикл игры, представляет собой определенный тип циклической обработки инструкцию (например, while цикл), которая работает с очень высокой частоты, например, 30 или 60 *кадров в секунду*.

Ниже приведен простой цикл игры:

![](images/image1.png "Это диаграмма простого цикла игры")

Технологии, которые рассмотрены ниже сделают абстрактными фактическое цикл while, но несмотря на Эта абстракция будет присутствовать концепцию обновления каждого кадра.

Производительность кода можно имеют более высокий приоритет в даже самая простая из игры. Например: функция, которая занимает 10 миллисекунд для выполнения может иметь значительное влияние на производительность игры — особенно в том случае, если он вызывается более одного раза для каждого кадра. Если игры в 30 кадров в запущена во-вторых затем это означает, что необходимо выполнить каждого кадра в миллисекундах, в разделе 33. В отличие от этого такая функция может даже не заметной, если выполняется только в ответ на нажатие кнопки в приложении не игры.

Распространенных типов логики, который может быть выполнено каждого кадра:

- **Чтение входных** – игра может потребоваться проверить, если он взаимодействовал с игрой путем проверки входных оборудования, например сенсорный экран, клавиатуры, мыши или игровой контроллер.
- **Перемещение** — какие перемещения из одного места в другое, обычно переместит очень маленьких объектов amount каждый кадр для создания иллюзии плавность движения.
- **Конфликт** – многие игры требуют частого тестирования ли перекрывающиеся или пересекающиеся различных объектов. Мы обсудим конфликтов более подробно в следующем разделе этой статьи. Перемещение и расчет столкновений могут обрабатываться системой выделенный физический моделирования.
- **Проверка наличия условиями играми** — состояние игры может управляться определенных условиях, такие как ли игрок, имеющем достаточное количество точек или ли истекло отведенное время.
- **Поведения искусственного Интеллекта** — логики каждый кадр, который может использоваться для управления поведением объектов, которые не контролируются проигрывателем, например patrolling злоумышленником или перемещение сопернику драйверы вокруг Овальный.
- **Подготовка к просмотру** — большинство игр будет обновлять отображаемые на экране каждого кадра. Это может быть выполнено в ответ на изменения, которые влияют на игры (например, символ, перемещение по уровня) или просто для предоставления визуальной отделки (например падающим снег или анимированных значков).

Имейте в виду, что многие из перечисленных выше действий можно изменять состояние приложения, в то время как многие приложения не игры, как правило, чтобы изменить состояние в ответ на возникновение события.


## <a name="content-loading-and-unloading"></a>Содержимого загрузки и выгрузки

Содержимое вручную Загрузка и выгрузка (или уничтожения) может потребоваться в зависимости от того, какую технологию вы используете в разработке. Вручную загрузку и выгрузку активов, может потребоваться по ряду причин:

 - Ресурсы может занять много времени для загрузки по отношению к длину один кадр. Некоторые ресурсы даже может занять секунды для загрузки, которые серьезно нарушило бы работу Если загружен mid игровой процесс. Если время загрузки особенно продолжительных (например, более чем на секунду или две) может потребоваться отобразить анимированный индикатор выполнения и экран загрузки.
 - Ресурсы могут использовать много оперативной памяти, требуя active управление что загружается в соответствии с размерами предлагает игры целевых платформ.
 - Игры может потребоваться отобразить несколько ресурсов, чем может поместиться в оперативной памяти. «Открыть World» игры часто включают в себя крупных средах какие игроков можно легко перемещаться по — это с экранами не загрузка. В этом случае может потребоваться создание пользовательской системы для потоковой передачи содержимого в и управление использования памяти.

Пользовательские форматы файлов может потребоваться обработка во время загрузки, требуя загрузки пользовательского кода.


## <a name="math"></a>Математический

Многие игры требуют более сложных математических операций, чем не игровых приложений. Само собой уровень math зависит от сложности игры. В целом трехмерных игр требуют дополнительные математические, чем двухмерные. К счастью, вы всегда можете начать работу с простых игр и курсы. Разработка игр может быть отличный способ обучения math!

Если вы знакомы с декартовой —, использующего координаты X и Y для размещения объектов —, а затем вы знаете достаточно, чтобы приступить к работе с разработки игр. Ниже показан декартовой положительный Y на рисунке указывает на вверх.

![](images/image2.png "Это показывает декартовой положительный Y на рисунке указывает на вверх")

Тригонометрические функции (например, синуса и косинуса) обычно используются в двухмерных игр, которые реализуют никакой формы поворота.

## <a name="content-pipelines"></a>Конвейерами содержимого

Термин *конвейера содержимого* — это процесс, который принимает файл для получения из формата (например, изображений PNG-файл) при в окончательный формат при использовании в игре. Конечный формат зависит от того, на котором тип содержимого, используется а также что технология используется для представления содержимого.

Некоторые содержимого конвейеры могут выполняться очень быстро и требуют никаких ручных действий. Например большинство игровых движков и API-интерфейсы можно загрузить в формате .png в необработанном формате. С другой стороны сложнее форматов (например, трехмерные модели) может потребоваться для обработки в другой формат до загрузки, и эта обработка может занять некоторое время в зависимости от размера и сложности актива.


# <a name="game-development-teams"></a>Группы разработчиков игр

Разработка игр вводит новые роли и заголовки для частных лиц, участвующих в процессе. Большинство разработчиков игр не смогут удовлетворить широкий набор навыков, необходимых для выпуска полной версии игры, поэтому существует ряд дисциплины. Имейте в виду, что это не полный список областей разработки — лишь некоторые из наиболее распространенных проблем.

- **Программист** — большинство пользователей, читающих в этой статье будет попадают в эту категорию. Роль программистом в разработке игр аналогична роли программиста в приложении не игры. Обязанности входит в написании логики для управления потоком игры, Разработка систем общие задачи в контексте конкретного проекта, добавления и отображения содержимого и – само собой — исправления ошибок.
- **2D исполнителя** — 2D исполнители отвечают за создание *2D активы*. К ним относятся файлы изображений для графического пользовательского интерфейса игры, примитивы, сред и символов. Если вы разрабатываете игру 3D, то 2D исполнители может не иметь ответственность за сред и символов. Вы найдете бесплатные картинки для игры в [ http://opengameart.org/ ](http://opengameart.org/) .
- **3D исполнители** — 3D исполнители отвечают за создание *трехмерных ресурсов*. К ним относятся трехмерных моделей для сред, символов и свойств (мебель растения и других неодушевленный). Некоторые команды отличить 3D исполнителей и 3D мультипликаторы зависимости от размера группы. Вы найдете бесплатные трехмерной графикой для игры в [ http://opengameart.org/ ](http://opengameart.org/) .
- **Конструктор игр** — игры конструкторы несут ответственность за определение того, как игры. Это могут быть высокоуровневые решений, таких как параметр игры, общую цель игры, и как проигрыватель они проходят игру. Игры конструкторы также может быть участвующих в очень подробные решения, такие как сопоставление входных данных для действия, определение коэффициенты для перемещения или уровень ИБП и проектирование уровня макета. Имейте в виду, что термин *конструктор* могут ссылаться на игры конструктор или визуальный конструктор, в зависимости от контекста.
- **Звуковые конструктор** — звуковой конструкторы отвечают за аудио ресурсам игры. Некоторые команды могут различать сотрудников, ответственных за создание звуковых эффектов и биографии, хотя небольшие команды, возможно, отдельный пользователь отвечает за все аудио.


# <a name="creating-a-game-idea"></a>Создание игр идея

Разработка игры может использоваться легко делать – в конце концов единственное требование не является «убедитесь, что-нибудь забавный.» К сожалению многие разработчики оказываются в убыток когда придет время для создания идею, из которого будет производиться запуск разработки.

Дисциплину проектирование игр не приводится легко и требуется способ улучшить так же, как искусство или программирования, но в этом разделе помогут вам начать работу путь.

Начинающим разработчикам следует начать с малого. Он может быть трудно не поддаться соблазну переделка больших, современных компьютерных игр, но меньшего размера игры может быть лучшую работу окружения обучения и быстрое развитие делает для более эффективной работы.

Многие игры, как для целей обучения, а также коммерческие игры, начать как улучшения или изменения существующих игр. Просмотрите другие игры для вдохновения — один из способов создания идеи. Например можно игры, в которой вы лично, например и попытайтесь определить какие характеристики игры обеспечивают весело. Возможно, просмотр, трудно переоценить важность мастерского механизм игры или проходит истории. Не забудьте при поиске новые идеи, рассмотрите возможность также «ретро» игры.

Другим способом создания новых идей всего определенного жанра, например головоломки игры, стратегии или platformers. Genre, знакомые разработчикам предоставит хорошей отправной точкой.

Изменяют существующий игры аналогична также образовательных, несмотря на то, что это может ограничить коммерческих жизнеспособности готового продукта. Процесс создания игры, хотя бы один, который является точным клоном, предоставляет полезные возможности для образования.


# <a name="game-development-technology"></a>Технологии разработки игр

Разработчики, использующие Xamarin.Android и Xamarin.iOS имеют широкий спектр технологий, доступных для помощи в разработке игр. В этом разделе будут рассматриваться некоторые наиболее популярные кроссплатформенных решений.


## <a name="cocossharp"></a>CocosSharp

CocosSharp является версией открытым исходным кодом, кросс платформенных игр 2D подсистема Cocos. Ядро предоставляет доступ к Android, iOS, Mac OS X, Windows Desktop, Windows RT и Windows Phone.

CocosSharp посвящена программисту простой API для разработки двухмерных игр. Рост в играх на мобильных устройствах помогла reignite популярность 2D разработки игр, делая CocosSharp жизнеспособную технологию хобби и коммерческих проектов одинаково. Предоставляется в виде исходного кода или DLL-файлы (которые можно получить с помощью NuGet), но он не предоставляет визуальный редактор; Таким образом любое взаимодействие с подсистемой CocosSharp требует знания по программированию.

Чтобы приступить к работе с CocosSharp, ознакомьтесь с нашей [CocosSharp направляющие](~/graphics-games/cocossharp/index.md).

Игра почему-то сердиты ниндзя создается с помощью CocosSharp, и это может быть хорошей отправной точкой, если вы ищете игру уже выполняемым для нескольких платформ:

## <a name="monogame"></a>MonoGame

MonoGame является открытым, кросс-версии платформы корпорации Microsoft XNA API. MonoGame можно использовать для создания игр для iOS, Android, Mac OS X, Linux, Windows, Windows RT, PS4, PSVita, Xbox One и коммутатора.

В отличие от CocosSharp MonoGame является технически не игровое ядро, но вместо этого интерфейс API разработки игр. Это означает, что для работы с MonoGame требуются непосредственно управления игровым объектам, вручную графических объектов и реализация общих объектов, таких как камеры и *сцены графы* (иерархии между игровым объектам). Чтобы понять различие, рассмотрим, что CocosSharp построена на основе MonoGame. MonoGame обобщает технологий платформы, такие как графики, отрисовки и аудио, хотя CocosSharp предоставляет код для организации и реализации логику игры.

MonoGame не поддерживает стандартной visual среды разработки, поэтому для работы с MonoGame требуются знания программирования.

Дополнительные сведения об использовании Sprite Kit см. запись блога:  [http://blog.xamarin.com/make-games-with-xamarin.ios-and-sprite-kit/](http://blog.xamarin.com/make-games-with-xamarin.ios-and-sprite-kit/)
- **Scene Kit** — Xamarin также обеспечивает поддержку платформы Scene Kit Apple, что упрощает реализации трехмерной графики в приложения iOS. Комплект для сцены также является технологией, предоставляемых компанией Apple, поэтому он содержит рекомендации по конкретной платформы, упомянутых выше Sprite Kit и интеграции. Дополнительные сведения о Scene Kit см. запись блога: [http://blog.xamarin.com/3d-in-ios-8-with-scene-kit/](http://blog.xamarin.com/3d-in-ios-8-with-scene-kit/)
- **OpenTK —** OpenTK (предназначенную для открытых Tool Kit) предоставляет низкоуровневый доступ OpenGL для Mac, iOS и Apple оборудования. Дополнительные сведения о OpenTK см. на главной странице:  [http://www.opentk.com/](http://www.opentk.com/)
