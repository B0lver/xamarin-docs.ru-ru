## <a name="this-class-is-not-key-value-coding-compliant-for-the-key-xxxx"></a>Этот класс не является значение ключа кодирования совместимое ключа XXXX

При появлении этой ошибки при загрузке файла NIB означает, что значение XXXX не найден управляемого класса. Это означает отсутствие объявления следующим образом:

```csharp
[Connect]
TypeName XXXX {
   get {
       return (TypeName) GetNativeField ("XXXX");
   }
   set {
       SetNativeField ("XXXX", value);
   }
}
```

Выше определения автоматически создается средой Visual Studio для Mac для всех файлов XIB, добавлять в Visual Studio для Mac в `NAME_OF_YOUR_XIB_FILE.designer.xib.cs` файла.

Кроме того, типов, содержащих этот код должен быть подклассом [NSObject](https://developer.xamarin.com/api/type/Foundation.NSObject/).  Если содержащий тип в пространстве имен, должна быть [[регистрация]](https://developer.xamarin.com/api/type/Foundation.RegisterAttribute/) атрибут, который предоставляет имя типа без пространства имен (как построитель интерфейс не поддерживает пространства имен в типах):

```csharp
namespace Samples.GLPaint {

    // The [Register] attribute overrides the type name registered
    // with the Objective-C runtime, in this case removing the namespace.
    [Register ("AppDelegate")]
    public class AppDelegate {/* ... */}
}
```

public partial class MyImageView : UIView {

## <a name="systemmissingmethodexception-no-constructor-found-for-foobarctorsystemintptr"></a>System.MissingMethodException: Не удалось найти конструктор для Foo.Bar::ctor(System.IntPtr)

Эта ошибка возникает во время выполнения, когда код пытается создать экземпляр классы, на который существует ссылка из файла интерфейс построителя. Это означает, что вы не добавляли конструктор, принимающий один IntPtr в качестве параметра.

Конструктор с дескриптором IntPtr используется для привязки объектов, управляемых с помощью представлений в неуправляемом коде.

Чтобы устранить эту проблему, добавьте следующий код в класс Foo.Bar:

```csharp
public Bar (IntPtr handle) : base (handle) { }
```
## <a name="type-foo--does-not-contain-a-definition-for-getnativefield-and-no-extension-method-getnativefield-of-type-foo-could-be-found"></a>Тип {Foo} не содержит определение для `GetNativeField' and no extension method `GetNativeField "типа может быть найден {Foo}

При появлении этой ошибки в конструкторе созданные файлы (*. xib.designer.cs), это означает, что одно из следующих действий:

 **(1) отсутствует разделяемого класса или базового класса**

Разделяемые классы, автоматически созданный конструктором должен иметь соответствующий разделяемых классов в пользовательском коде, которые наследуют некоторые подкласс `NSObject`, часто `UIViewController`. Убедитесь, что имеется такой класс для типа, который дает ошибку.

 **(2) пространства имен по умолчанию изменено**

Файлы конструктора создаются с помощью параметров пространства имен по умолчанию проекта. Если после изменения этих параметров, или переименовать проект, созданный разделяемые классы больше не может быть в том же пространстве имен, совпадают с аналогичными пользовательского кода.

Параметры пространства имен можно найти в диалоговом окне Параметры проекта. Пространство имен по умолчанию находится в **Общие -> Параметры Main** раздела. Если поле пусто, имя проекта используется по умолчанию. Дополнительные параметры пространства имен можно найти в **исходный код -> политики именования .NET** раздела.

## <a name="ran-out-of-trampolines-of-type-0"></a>Вышел за пределы trampolines типа 0

Если получить это сообщение во время выполнения устройства, можно создать дополнительные тип 0 trampolines (ОПРЕДЕЛЕННОГО типа), изменив параметры вашего проекта, раздел «iPhone построения».  Вы хотите добавить лишние аргументы для устройства построения целевых объектов:

 `-aot "nrgctx-trampolines=2048"`

Количество trampolines по умолчанию — 1024.  Попробуйте увеличить это число, пока не будет достаточно для использования универсальных шаблонов.

## <a name="debugger-is-unable-to-connect-with-the-device"></a>Отладчику не удается подключиться к устройству

Существует несколько причин, по которым отладчик не сможет подключиться к приложению, в зависимости от режима, используется для подключения (USB или Wi-Fi).

 **Если устройство и узел отладчика, находятся в разных сетях**, брандмауэр или частную сеть может препятствовать приложения из подключения к узлу отладчик в режиме Wi-Fi.

 **Visual Studio для Mac не могут иметь возможность направлять запрос правильный IP-адрес узла**. В Wi-Fi режим Visual Studio для Mac предоставляет приложению все IP-адреса, он может найти узла и приложение попытается их все для просмотра, если она может использовать любой из них для подключения к Visual Studio для Mac.

 **Другое устройство подключено к USB-порту на узле.** В некоторых случаях с другими устройствами, подключенными к USB-ПОРТУ портов на узле известно, к каким-либо образом повлиять на отладку в режиме USB.

Если режим Wi-Fi или USB не работает, можно легко попробовать другой: Откройте в Visual Studio для Mac параметров, перейдите на страницу отладчик отладчик/параметры/iPhone и переключать флажок «Debug устройства iOS по Wi-Fi, а не по USB».   Если не работает, вы увидите Дополнительные сведения об ошибке в консоль устройства в подробном режиме (которой включается путем добавления «-v - v - v» на mtouch дополнительные аргументы в параметрах проекта).

## <a name="uialertview-or-uiactionsheet-vanish-after-being-created"></a>UIAlertView или UIActionSheet удалить после создания

Это значение не обязательно совпадают с значение версии пакета SDK — это изменение отразится на минимальную версию операционной системы будет установить приложения, который может быть более ранней, чем пакет SDK, а используйте только те API, которые существуют в старой операционной системе или предотвратить использование более новыми функциями, проверка версии операционной системы среды выполнения с помощью Ks.
